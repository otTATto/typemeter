# ドキュメント作成に関するルール

## 全般

- プロジェクトに関するドキュメントは `docs/` 以下に Markdown ファイルとして格納される
  - 階層構造を意識すること
    - 1 つのドキュメントは、1 つのテーマのみを扱う
    - 1 つの見出しで、1 つのサブテーマを扱う
    - 1 つの段落で、1 つの話題を扱う
- 書くべきことは明瞭かつ簡潔に書き、書かなくてよいことは一切書くべきでない
  - 他の場所ですでに書いている内容は重複して書かずに、参照したり最低限の引用に留めたりする
- 結論を先行させ、過程を後続させる
  - 全体から部分、概要から具体、既知から未知へと展開させる
  - 現状どうなっているか、結局なにをすべきかという結論がもっとも重要なので、なるべく先行させる
  - その理由や過程、具体的な方法なども重要だが、これらは後述で構わない
- 用語や知識は厳格に
  - 表記揺れしてはならない（はじめは正式名称を出し、「以下、○○という。」と宣言する）
  - 断定した知識には根拠を示す
- 見出しや段落の一行目は、続く内容の適切な要約になっているべきである
  - 一行目として、その段落で言いたいことや主張を先行させる
  - その理由や詳しい説明は二行目以降に書かれるか、長い場合は段落や見出しすらわけて後述する
- 常にそのドキュメントを初めて読む人のことを考えるべきである
  - これまでの経緯や暗黙知を前提にしてはならない
  - 後方互換性のために（過去にそのドキュメントを読んだ人の認知不可を下げるために）ドキュメントの変更を最小限に留めようとするべきではない

## 各ドキュメントの書き方について

- 各ドキュメントは、その冒頭で以下のような内容を明示する（人間は読むのに時間がかかるため、読者がそのドキュメントを本当に読むべきかどうかをすぐに判断できるようにしてあげるべきである）
  - そのドキュメントに何が書かれているか（項目を列挙するだけでなく、できれば簡単な要約も含意させる）
  - そのドキュメントに何が書かれていないか（書かれている別なドキュメントがあれば具体的に参照すること）
  - そのドキュメントを読むべき人・読むべき状況はどういうものだと想定されるか（ただし特別に限定しにくい場合は書かなくて良い）
- 各ドキュメントは、複数の見出しから構成される
  - 各見出しは、複数の段落から構成される
  - 各見出しセクションは、その始めと終わりとに、それぞれ同じ内容の「この見出しで言いたいこと」が書かれる（つまり、見出し→主張→段落1→段落2→…→主張（結論）、という構造）
- まずは現状の整理に焦点を当て、それに至った経緯は後述する（時系列に書くことが悪いことではないが、結論を先行させ、多くの読者が得たい情報になるべく速く届かせてあげるべきである）
  - 現状の整理には、表や Mermaid 記法を活用し、複雑な状況をなるべくわかりやすく見通せるよう努める
  - 経緯を説明する場合は、基本的には時系列昇順に、いつ・誰が・どういう理由や背景で・何をしたのかを簡潔にする
- 表や図内、箇条書きなどを除いて、すべて敬体（です・ます調）で統一する

以下はまだルールとして整備途中です。今後、さらに追加されたり、変更されたりするかもしれません。

あくまでも例であり、必ず従うべきテンプレートではありませんが、読み手の認知不可を下げるため、特別な理由が無い限りは共通しているべきです。

```md
# タイトル

## 概要

このドキュメントでは、○○について書いています。具体的には、以下のようなことについて書いています。

- 書いている内容①
- 書いている内容②

ただし、以下のような内容は扱っていません。

- 書いていない内容①（詳しくは、○○を参照してください）
- 書いていない内容②（現在、参照するべきドキュメントはありません）
```
