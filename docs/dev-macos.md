# macOS での開発・検証の注意点

## 概要

このドキュメントでは、macOS 上で typemeter を開発・検証する際に固有の注意点について書いています。具体的には、以下のようなことについて書いています。

- 再ビルドするたびに入力監視の権限が「オンに見えるのに効かない」状態になる仕組みと、その恒久対策（自己署名証明書）
- 動作検証時にハマりやすい落とし穴（残留プロセス・ノッチ）

ただし、以下のような内容は扱っていません。

- アプリの常駐・ライフサイクルの仕様（詳しくは、[app-lifecycle.md](./app-lifecycle.md) を参照してください）
- 一般的なビルド手順（詳しくは、リポジトリルートの `README.md` を参照してください）

macOS でビルド・動作検証を行う人が読むことを想定しています。

## 入力監視権限とコード署名

macOS は入力監視（Input Monitoring）の許可を**アプリのコード署名に紐づけて**管理しているため、署名が変わる再ビルドを行うたびに権限が実質的に失効します。恒久対策は、ビルドをまたいで同一になる自己署名証明書で署名することです。

失効時の症状は次のとおりで、非常に気づきにくいものです。

- システム設定上は typemeter が入力監視の一覧に**オンのまま並んで見える**
- エラーやプロンプトは一切出ず、キーストロークのイベントだけが黙って届かなくなる
- Finder / Dock 経由で起動した場合のみ発生し、ターミナルからバイナリを直接実行（`/Applications/typemeter.app/Contents/MacOS/typemeter`）するとカウントされる、という切り分けが可能

原因は、署名なしビルドに施される ad-hoc 署名がビルドのたびに変わり、権限を許可した時点のバイナリと別物と判定されるためです。その場しのぎとしては「入力監視の一覧から typemeter を削除 → アプリ再起動 → 再許可」で復旧しますが、再ビルドのたびに繰り返すことになります。

恒久対策として、開発機で自己署名証明書を作り、常にそれで署名してビルドしてください。

1. キーチェーンアクセス > 証明書アシスタント > 証明書を作成 を開き、名前 `typemeter-dev`・証明書のタイプ「コード署名」で自己署名証明書を作成する
2. ビルド時に署名 identity を指定する

   ```sh
   APPLE_SIGNING_IDENTITY="typemeter-dev" bun run tauri build
   ```

3. この方法での初回ビルド後に一度だけ入力監視を再許可する。以降は再ビルドしても署名が同一のため、権限が維持される

## 検証時の落とし穴

常駐アプリ化以降、macOS での動作検証には「見えない状態で生き続けるもの」に起因する罠が 2 つあります。

**残留プロセス**: ウィンドウを × で閉じたインスタンスは、Dock からも消えた状態で生き続けています。この状態で新しいビルドを起動しても、単一インスタンス機構が既存プロセスへ転送して新プロセスは即終了するため、**検証しているつもりのバイナリが動いていない**ことがあります。検証前に必ず残留プロセスを終了してください。

```sh
killall typemeter
```

**ノッチ**: ノッチ付きの MacBook では、メニューバーのアイコンが多いと新しいアイコンがノッチ領域に隠れて完全に見えなくなります。トレイアイコンが表示されないように見えたら、まず他のメニューバーアイコンを減らすか、外部ディスプレイで確認してください。

いずれも「壊れているように見えて実は環境要因」というパターンなので、macOS で不可解な挙動に遭遇したら、コードを疑う前にこの 2 点と前節の権限失効を確認するのが近道です。
